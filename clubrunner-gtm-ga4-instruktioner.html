Varför ClubRunner kräver särskild konfiguration

ClubRunner använder fem separata domäner. Utan rätt konfiguration tolkar GA4 navigering mellan domänerna som externa besök och skapar nya sessioner med felaktiga trafikkällor. Alla fem måste behandlas som interna.

Publik sajt (alias)rotary[distrikt].se
Portal / CMS-backendportal.clubrunner.ca
Innehålls-APIcontent.clubrunner.ca
Admin-gränssnittadmin.clubrunner.ca
Medlems- och klubbdatacrsadmin.com

Filer som besökare laddar ned via ClubRunner hämtas dessutom från en separat blob-domän: clubrunner.blob.core.windows.net. Det kräver en egen trigger (se avsnitt 2g).

Ställ om GTM och GA4 till engelska

Alla menyval och fältnamn i dessa instruktioner är på engelska, exakt som de visas i gränssnitten. Följ stegen nedan för att säkerställa att du ser samma benämningar.

Google Tag Manager — lägg till ?hl=en i adressfältet:
https://tagmanager.google.com/?hl=en

Google Analytics 4 — lägg till ?hl=en i adressfältet:
https://analytics.google.com/?hl=en

För ett distrikt är det nödvändigt att konfigurera spårningen, se supportsidan om Trafikanalys.

Gäller det här också för klubb-sajt?

Motsvarande konfiguration kan göras för en ClubRunner-klubb. Tekniken är identisk — domäner, triggers och taggar fungerar på samma sätt. Skillnaden är att en typisk klubb-sajt har så liten trafikvolym att statistiska mätningar sällan ger meningsfulla resultat: segmenteringar, funnel-analyser och trafikkällefördelningar kräver tillräckligt många observationer för att mönstren ska vara tillförlitliga.

Flera av de avancerade taggarna i detta dokument är däremot värdefulla oavsett volym, eftersom de inte bygger på statistik utan ger en direkt lista över konkreta händelser — även en handfull observationer är användbara:

Dokumentnedladdningar (2f/2g)Vilka protokoll, stadgar eller blanketter klubbmedlemmar faktiskt hämtar.
Sökordsspårning (2i)Vad besökare letar efter — avslöjar innehåll som saknas eller är svårt att hitta i menyn.

Utöver dessa ger följande övergripande nyckeltal alltid ett bra underlag, oavsett trafikvolym:

TrendSessioner och användare över tid — ser trafiken ut att växa eller minska?
InnehållMest besökta sidor — vilket innehåll lockar besökare?
TrafikkällorOrganisk sökning, direkttrafik eller hänvisningar — hur hittar folk sajten?
EnhetstypMobil kontra dator — avgör om layouten fungerar för den faktiska publiken.
EngagemangAverage engagement time per session — stannar besökarna och läser?
AvvisningEngaged sessions-andel — andel besök där någon faktisk interaktion sker.

Rekommendation: sätt upp grundtaggen, dokumentnedladdning och sökordsspårning på en klubb-sajt oavsett volym — de kostar inget extra och ger direkt användbar data.

Events som skapas i GTM kan skicka med parametrar (t.ex. vilken URL som klickades). Parametrarna syns inte automatiskt i GA4-rapporter — de måste registreras som "custom dimensions" för att bli sökbara och rapporterbara. Rapporterna byggs i nästa avsnitt 1d.

Admin › Custom definitions › Create custom dimension

Skapa en rad per parameter:

Dimension name (valfritt) Scope Event parameter / User property Description (används av)

Link text Event link_text outbound_click, file_download (taggarna i 2f och 2g)

    2d

    Utgående länkklick — trigger och tagg

    Registrerar klick på externa länkar. ClubRunners egna domäner, bulletin-länkar och sid-filnedladdningar måste exkluderas så att de inte räknas som externa eller dubbelräknas mot nedladdningstaggarna.

    Steg 1 — Skapa trigger

    Triggers › New
    Trigger type: Click – Just Links This trigger fires on: Some Link Clicks // Villkor — triggar när ALLA är uppfyllda: // Gate: bara verkliga http(s)-länkar – stoppar filter-/JS-länkar utan href Click URL matches RegEx ^https?:// // Byt ut rotary[distrikt].se mot din publika domän: Click URL does not start with https://rotary[distrikt].se Click URL does not start with https://portal.clubrunner.ca Click URL does not start with https://content.clubrunner.ca Click URL does not start with https://admin.clubrunner.ca Click URL does not contain crsadmin.com Click URL does not contain clubrunner.blob.core.windows.net Click URL does not contain /bulletin/View/ Click URL does not contain view.officeapps.live.com Trigger name: Outbound link click
    Varför blob-domänen och Office Viewer exkluderas: Filer i "Dokument"-widgeten serveras från clubrunner.blob.core.windows.net och fångas redan av nedladdningstaggen i avsnitt 2g. Utan denna rad skulle samma klick även utlösa outbound_click och dubbelräknas som externt. view.officeapps.live.com läggs till som defensivt undantag: Word-, Excel- och PowerPoint-dokument i /documents/sv-se/ täcks redan av triggern i 2f (ClubRunners href pekar alltid på rotary2405.se och omdirigeringen sker server-side), men om en länk av misstag hårdkodas direkt till Office Viewer fångas den ändå inte som outbound_click.
    Viktigt: Varje rad är ett eget villkor. Klicka "+ Add condition" för varje ny rad. Alla villkor i listan måste uppfyllas samtidigt för att triggern ska aktiveras — det är korrekt beteende här.
    Varför RegEx-raden behövs: En trigger som bara har negativa villkor ("does not…") aktiveras felaktigt av länkar utan riktig href — t.ex. ClubRunners "Filter"-knapp på evenemangssidan, som filtrerar via JavaScript. Då blir {{Click URL}} tom, och en tom sträng uppfyller alla negativa villkor samtidigt. Villkoret Click URL matches RegEx ^https?:// kräver en verklig http(s)-adress och släcker den falska träffen. (Triggrarna 2e–2g drabbas inte, eftersom de bygger på positiva contains/starts with-villkor.)

    Steg 2 — Skapa tagg

    Tags › New
    Tag type: Google Analytics: GA4 Event Configuration: [välj din Google tag] Event name: outbound_click Event parameters: Parameter name: link_url Value: {{Click URL}} Parameter name: link_text Value: {{Click Text}} Trigger: Outbound link click Tag name: GA4 – Outbound link click

    Dokumentnedladdning — trigger och tagg

    Fångar klick på ClubRunners dokumentlänkar (filer i Dokument). Matcha på sökvägen /documents/sv-se/ i URL:en i stället för filtypsändelse — ClubRunners dokumentlänkar har inte alltid en filändelse, och sökvägsmatchning är enklare och mer robust.

    Steg 1 — Skapa trigger

    Trigger type: Click – Just Links This trigger fires on: Some Link Clicks Click URL contains /documents/sv-se/ // Matchar ClubRunners dokumentsidor – enklare och mer specifik än RegEx Trigger name: Document download
    Täcker automatiskt Office-dokument: ClubRunner använder samma URL-mönster (/documents/sv-se/) i href-attributet för alla dokumenttyper — PDF, Word, Excel och PowerPoint. Omdirigeringen till Microsoft Office Online Viewer (view.officeapps.live.com) sker server-side efter klicket; GTM fångar href-attributet före omdirigeringen. Ingen separat trigger behövs för Office-dokument. Händelser som syns i GA4 som utgående klick till view.officeapps.live.com är en artefakt av Enhanced Measurement och försvinner när Outbound clicks stängs av (avsnitt 1b).

    Steg 2 — Skapa tagg

    Tag type: Google Analytics: GA4 Event Configuration: [välj din Google tag] Event name: file_download Event parameters: Parameter name: file_name Value: {{Click URL}} Parameter name: link_text Value: {{Click Text}} Trigger: Document download Tag name: GA4 – Document download
    2g

    Nedladdning av sidfiler via ClubRunner blob-lagring — trigger och tagg

    Filer som är uppladdade i widgeten Download Page Files lagrar ClubRunner på en separat domän: clubrunner.blob.core.windows.net. Dessa filer ligger under ett sökvägsprefix som inleds med distriktets/klubbens ClubRunner-ID (t.ex. /00000060099/). Eftersom det ID:t är unikt för distrikt/klubb räcker det som matchning — triggern blir då oberoende av språk- och mappstruktur. De kräver en separat trigger eftersom domänen skiljer sig från din publika sajt.

    Blob-domänen måste också vara med i undantagen i "Outbound link click"-triggern (avsnitt 2d) så att dessa nedladdningar inte samtidigt räknas som externa klick.

    Vad är skillnaden mot 2f? Båda triggrarna matchar på sökväg, inte filtyp. 2f fångar dokumentlänkar på din egen domän (/documents/sv-se/), medan 2g fångar filer som ClubRunner serverar från blob-lagringen på en annan domän (clubrunner.blob.core.windows.net/[org-ID]/). De ligger på olika domäner och behöver därför var sin trigger.

    Steg 1 — Skapa trigger

    2h

    Konsolidera sidvisningar: omdöpta sidor och skiftlägeskorrigering (canonical-sökväg)

    GA4 grupperar sidvisningar på page_location (den faktiska URL:en). På ClubRunner gör det att samma sida ofta splittras på flera rader av två skäl: omdöpta sidor nås via en gammal alias-URL parallellt med den rena adressen, och skiftläge i sökvägen (/Events vs /events, /ClubInfo/… vs /clubinfo/…) skapar dubbletter. Variabeln nedan åtgärdar båda.

    När en sida byter namn låter ClubRunner den gamla adressen fortsätta fungera — men utan 301-omdirigering. Innehållet serveras på en lång alias-URL, t.ex. /60099/Page/Show?ClassCode=Page&Slug=…, parallellt med den rena /page/<slug>. Resultatet blir två sidor i rapporten för samma innehåll.

    Två fällor: (1) alias-URL:en 301-omdirigerar inte — den svarar 200 — så GA4 registrerar båda adresserna som varsin sida. (2) canonical-taggen på alias-sidan pekar mot fel värd: portal.clubrunner.ca i stället för den publika domänen. Kopierar du därför canonical rakt av till page_location får du en portal.clubrunner.ca-rad i stället.

    Så här ser det faktiskt ut för samma sida nådd via två ingångar:

    Ingång (URL du klickar) Canonical som ClubRunner skickar
    rotary2405.se/page/support-logga-in-hjälp https://rotary2405.se/page/support-logga-in-hjälp ✅
    rotary2405.se/60099/Page/Show?…&Slug=support-logga-in-hjälp https://portal.clubrunner.ca/page/support-logga-in-hjälp ❌ fel värd

    Poängen: sökvägen /page/support-logga-in-hjälp är identisk i båda canonical — det är bara värden som skiljer. Därför bygger variabeln page_location av canonical-sökvägen (rätt i båda fallen) men sätter den alltid på den publika domänen via location.origin.

    Steg 1 — Variabel som bygger ren page_location

    Variables › New › Custom JavaScript
    Variable type: Custom JavaScript Variable name: Canonical Page Location function() { var keep = ['utm_source','utm_medium','utm_campaign','utm_term', 'utm_content','utm_id','gclid','gbraid','wbraid', 'dclid','gclsrc','fbclid','msclkid','srsltid']; var path; try { var c = document.querySelector('link[rel="canonical"]'); path = c && c.href ? new URL(c.href).pathname : location.pathname; } catch (e) { path = location.pathname; } path = path.toLowerCase(); // ClubRunner är skiftlägesokänsligt – slår ihop /Events, /ClubInfo/… osv if (/^\/page\/showhomepage\/?$/.test(path)) path = '/'; // hemsidans interna route → roten var loc = location.origin + path; // publik domän + canonical-sökväg var src = new URLSearchParams(location.search); var out = new URLSearchParams(); keep.forEach(function(k){ if (src.has(k)) out.set(k, src.get(k)); }); var qs = out.toString(); return qs ? loc + '?' + qs : loc; }

    Canonical-sökvägen (omdöpta sidor). Variabeln tar sökvägen ur canonical-taggen (samma rena /page/<slug> oavsett om besökaren kom in via alias-URL:en eller den rena adressen), sätter den på den publika domänen via location.origin, och behåller bara kampanjparametrar (utm_*, gclid m.fl.) så att attributionen överlever.

    Skiftlägeskorrigering. Raden path = path.toLowerCase() gör sökvägen gemen — ClubRunner är skiftlägesokänsligt, så /Events, /ClubInfo/… och /ClubsMap slås ihop med sina gemena varianter (i ett års data för rotary2405.se låg ~1 466 visningar fördelade på 184 sådana dubblettgrupper, varav /clubinfo/* ofta var splittrade i tre skiftlägesvarianter). Lowercasingen normaliserar samtidigt %-kodningens skiftläge (%C3%A4 → %c3%a4). Eftersom bara sökvägen görs gemen påverkas inte kampanjparametrarnas värden.

    Förutsättning för lowercasing: detta är säkert just för att ClubRunner serverar samma sida oavsett skiftläge i URL:en. På en server där sökvägar är skiftlägeskänsliga skulle toLowerCase() kunna slå ihop två olika sidor — då skulle inte den raden kunna användas.
    Specialfall hemsidan: ClubRunner serverar hemsidan på två live-adresser — den rena roten / och den interna routen /page/showhomepage (dit logotyp-/"Hem"-länkar ofta går). Båda har självrefererande canonical, så canonical ensam slår inte ihop dem; därför tvingar if-raden i variabeln /page/showhomepage till roten. Den bara routen /60099/Page/Show utan ?…&Slug= är däremot en felsida ("Kontot hittas inte", soft 404) — den ska inte normaliseras bort. Dyker den upp i rapporterna är det en nyttig signal om trasiga länkar (larma t.ex. på page_title som innehåller "General Error").

    Steg 2 — Skriv över page_location i Google-taggen

    Öppna Google-taggen från avsnitt 2b. Under Configuration settings (Shared event settings) lägg till en parameter:

    Configuration parameter: page_location Value: {{Canonical Page Location}}
    Effekt: Den automatiska page_view och alla GA4-event som ärver Google-taggen rapporteras nu på den rena sökvägen, oavsett vilken alias-URL besökaren kom in på. Eftersom kampanjparametrarna behålls påverkas inte trafikkällorna.
    2i

    Sökrutan i header — spåra sökord (site search)

    ClubRunners inbyggda sökruta (fältet med id="searchString" i sidhuvudet) skickar inte sökordet som en query-parameter (?q=…), vilket är vad GA4:s Enhanced Measurement förväntar sig. Istället navigerar ClubRunner till en sökväg på formen /Search/<sökord>. Det innebär att sökordet måste extraheras ur sökvägen med en Custom JavaScript-variabel.

    Bekräftat beteende: Innan URL-kodning ersätter ClubRunner tecknen . % ? : / + * & " med platshållarna -~0- till -~8-. Variabeln nedan återställer dessa till originaltecken.

    Steg 1 — Variabel som extraherar sökordet

    Variables › New › Custom JavaScript
    Variable type: Custom JavaScript Variable name: JS-sitesearch CR function() { var path = window.location.pathname; var match = path.match(/^\/[Ss]earch\/(.+)$/); if (!match) return undefined; var term = decodeURIComponent(match[1]); // Återställ ClubRunners platshållare till originaltecken (se doSearch()) var replacements = [ ['-~0-', '.'], ['-~1-', '%'], ['-~2-', '?'], ['-~3-', ':'], ['-~4-', '/'], ['-~5-', '+'], ['-~6-', '*'], ['-~7-', '&'], ['-~8-', '"'] ]; for (var i = 0; i < replacements.length; i++) { term = term.split(replacements[i][0]).join(replacements[i][1]); } return term; }

    Steg 2 — Trigger

    Triggers › New
    Trigger type: Page View This trigger fires on: Some Page Views Page Path matches RegEx ^/[Ss]earch/ Trigger name: CR-sitesearch

    Steg 3 — Tagg

    Tags › New
    Tag type: Google Analytics: GA4 Event Configuration: [välj din Google tag] Event name: view_search_results // GA4:s rekommenderade eventnamn för sökningar Event Parameters Parameter name: search_term Value: {{JS-sitesearch CR}} Trigger: CR-sitesearch Tag name: GA4 – sitesearch CR
    UI-fälla i eventSettingsTable: när du fyller i parameter-värdet, skriv {{ och välj variabeln ur listan som poppar upp — bekräfta med Tab eller klick utanför fältet. Om du klickar direkt från variabelfältet till värdefältet med musen nollställs valet.

    Special Event — utvalda länkar och knappar med egen etikett

    Utöver de generella kategorierna ovan (utgående klick, nedladdningar, sökning) vill man ibland följa enskilda, utvalda länkar eller knappar individuellt i rapporter — t.ex. en anmälningslänk — med en läsbar etikett i GA4 (t.ex. "Intresseanmälan") i stället för bara länkadressen. Ett gemensamt eventnamn, special_event, används för alla dessa, oavsett hur många olika länkar/knappar som spåras. Nya specialevent kräver normalt ingen ny trigger eller tagg — bara en radändring i en av två delade komponenter. På ClubRunner webbsidor finns länkarna i tre huvudtyper, beroende på vilken widget som används för länken:

    ClubRunner-widget Vad som går att redigera Matchas via
    Textwidget (länk i löpande text) Bara länktext, ingen HTML URL-lista (Special Event – Link List)
    Knapp-widget ("Bara en knapp") Knapptext och länkadress, ingen HTML URL-lista (Special Event – Link List)
    Custom widget Fri HTML, inkl. attribut data-ga-label-attribut (Special Event – Attribute)
    Knapp-widgeten lägger till ett efterföljande mellanslag i länkadressen (t.ex. href="https://exempel.se "), vilket gör att ett exakt equals-villkor missar träffen. RegEx-matchningen som används i konfigurationen nedan kräver ingen exakt slutmatchning och påverkas därför inte av mellanslaget.
